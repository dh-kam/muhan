# Coding Guidelines (Andrej Karpathy 기반)
# Source: https://github.com/multica-ai/andrej-karpathy-skills
#
# LLM 코딩 실수를 줄이기 위한 행동 가이드라인.
# 속도보다 신중함에 편향. 사소한 작업에는 판단력을 사용할 것.

## 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전:
- 가정을 명시적으로 진술할 것. 불확실하면 질문할 것.
- 여러 해석이 존재하면 제시할 것 — 조용히 하나를 선택하지 말 것.
- 더 단순한 접근법이 존재하면 말할 것. 필요하면 반대 의견을 제시할 것.
- 불명확한 것이 있으면 멈출 것. 혼란스러운 부분을 명명하고 질문할 것.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

- 요청된 것 이상의 기능 금지.
- 단일 사용 코드에 대한 추상화 금지.
- 요청되지 않은 "유연성"이나 "설정 가능성" 금지.
- 불가능한 시나리오에 대한 에러 처리 금지.
- 200줄로 작성했는데 50줄로 가능하면 다시 작성할 것.

자문: "시니어 엔지니어가 이것이 과도하게 복잡하다고 말할 것인가?" 그렇다면 단순화할 것.

## 3. 외과적 변경

**필요한 것만 수정할 것. 자기가 만든 지저분함만 정리할 것.**

기존 코드 편집 시:
- 인접 코드, 주석, 포맷팅을 "개선"하지 말 것.
- 깨지지 않은 것을 리팩터링하지 말 것.
- 자신이라면 다르게 했을지라도 기존 스타일을 따를 것.
- 관련 없는 데드 코드를 발견하면 언급만 할 것 — 삭제하지 말 것.

변경으로 인해 고아가 생겼을 때:
- 자신의 변경으로 사용되지 않게 된 import/변수/함수는 제거할 것.
- 기존에 존재하던 데드 코드는 요청받지 않는 한 제거하지 말 것.

테스트: 변경된 모든 줄은 사용자의 요청에 직접 추적 가능해야 한다.

## 4. 목표 주도 실행

**성공 기준을 정의할 것. 검증될 때까지 반복할 것.**

작업을 검증 가능한 목표로 변환:
- "검증 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시킬 것"
- "버그 수정" → "재현하는 테스트를 작성하고 통과시킬 것"
- "X 리팩터링" → "전후로 테스트가 통과하는지 확인할 것"

다단계 작업에는 간단한 계획을 진술:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
3. [단계] → 검증: [확인사항]
```

강한 성공 기준은 독립적 반복을 가능하게 한다. 약한 기준("작동하게 해")은 지속적 명확화가 필요하다.
